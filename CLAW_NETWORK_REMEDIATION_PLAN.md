# Claw Network 稳定性整改方案

## 1. 文档目的

本文档用于沉淀本次 `OpenClaw + claw-network` 故障的正式结论，并给出可执行的整改方案，目标如下：

- 明确本次故障的直接原因与根因
- 区分 `OpenClaw` 本体问题与 `claw-network` 集成问题
- 给出短期止血、中期加固、长期重构的路线
- 建立后续发布、迁移、巡检、回归验证的工程基线

本文档面向：

- `claw-network` 插件维护者
- `claw-network-release` 发布维护者
- OpenClaw 接入运维人员
- 后续接入该网络的第三方实例维护者

## 2. 事故摘要

本次故障表现为多台已接入 `claw-network` 的 OpenClaw 实例出现“龙虾不工作”或“突然断开”的现象。经排查，部分实例出现如下症状：

- `openclaw health` 无法执行
- OpenClaw Gateway 未启动
- `claw-network` 相关能力全部失效
- 已部署实例之间表现不一致，导致用户误以为是“网络整体故障”

实际情况并非 `claw-network` 整体后端立即不可用，而是多个实例本地配置进入非法状态，导致 OpenClaw 在启动前的配置校验阶段即失败。

## 3. 结论

### 3.1 结论一：不是 OpenClaw 本体缺陷导致

当前证据不足以证明 OpenClaw 核心运行时存在系统性缺陷。

已确认：

- OpenClaw CLI 本体在配置合法时可以正常运行
- Gateway、Health、插件装载能力在配置合法时可以恢复
- 本次故障的触发点出现在 `claw-network` 插件配置与安装产物不一致

因此，本次事故不应归因于 OpenClaw 本体，而应归因于 `claw-network` 插件及其安装链路。

### 3.2 结论二：线上故障由 release 产物触发，但根因来自插件源码设计

从影响面看：

- 线上受影响的是已部署实例与 `release` 安装产物

从根因看：

- 同类问题同时存在于 `openclaw-a2a-mvp` 与 `claw-network-release`
- `mvp` 中的安装逻辑、schema 定义、路径假设已存在缺陷
- `release` 将该缺陷带到了实际部署环境

因此，准确结论为：

- 线上故障由 `release` 安装产物触发
- 根因来自 `claw-network` 插件源码与安装链路的工程约束不足

## 4. 故障根因分析

## 4.1 直接原因：插件配置进入非法状态

已确认多个实例的 `~/.openclaw/openclaw.json` 中存在如下结构：

```json
{
  "plugins": {
    "entries": {
      "claw-network": {
        "enabled": true,
        "config": {
          "endpoint": "...",
          "runtimeId": "...",
          "name": "...",
          "ownerName": "...",
          "pythonBin": "...",
          "clientPath": "...",
          "dataDir": "...",
          "sidecarScript": "...",
          "onboarding": {}
        }
      }
    }
  }
}
```

其中 `onboarding` 会被部分运行时校验链路判定为非法附加字段，导致：

1. OpenClaw 读取配置
2. 插件配置校验失败
3. `openclaw health` 失败
4. Gateway 启动失败
5. 上层龙虾网络能力全部不可用

这意味着故障不是“某个工具调用失败”，而是“实例根本无法完成启动”。

## 4.2 根因一：配置 schema 存在多份定义，且未强制一致

本次排查中发现，`claw-network` 存在至少三类配置定义来源：

- `openclaw.plugin.json` 中的 `configSchema`
- `index.js` 中的运行时 `configSchema`
- `scripts/install_local.py` 中写入配置时的字段集合

问题在于：

- 这些定义并非单一事实来源
- 安装脚本会写入字段，但 manifest 未必声明
- 运行时读取逻辑可接受的字段，与安装脚本输出的字段不一定一致

这是本次事故的核心工程缺陷。

## 4.3 根因二：安装脚本缺少“产物校验”

安装脚本在写入 `openclaw.json` 后，并未执行以下关键校验：

- 生成配置是否完全符合 plugin manifest schema
- 目标 OpenClaw 版本是否接受该配置
- 使用该配置后 Gateway 是否可启动

结果是：

- 配置文件可以成功写盘
- 但写出的配置对目标运行时是非法的
- 故障被延迟到“下一次启动或重启 Gateway”时才爆发

这也是“过几天突然断掉”的根本原因之一。

## 4.4 根因三：路径与运行目录存在硬编码假设

插件运行时代码对工作目录存在写死历史路径的做法，例如默认假设工程位于：

- `/home/openclaw-a2a-mvp/...`

一旦项目从 `mvp` 迁移到 `release`，或部署目录变化，就可能出现：

- `clientPath` 已更新
- `dataDir` 已更新
- 但运行时 `cwd` 仍停留在旧工程目录

这类缺陷不会总是立即触发，但会在部署迁移、脚本更新、路径调整后积累风险。

## 4.5 根因四：缺乏版本化配置迁移机制

当前 `claw-network` 配置未形成明确的：

- `configVersion`
- migration 流程
- 旧字段淘汰策略
- 兼容性窗口

因此，当字段定义发生变化时，系统没有可靠方式自动将旧实例升级到合法状态，只能依赖手工修复。

## 5. 对业务与系统的影响

本次问题的工程影响大于单次可用性影响，原因如下：

- 用户会认为“龙虾网络随机失效”
- 同类实例在不同时间点会以不同方式失效
- 运维人员难以从用户反馈直接判断失败阶段
- 新接入用户缺乏稳定预期，降低系统可信度

如果不整改，后续很可能出现：

- 新发布版本再次写入非法配置
- 已部署实例在例行重启后集体掉线
- 迁移新目录或新部署方式时再次出现路径漂移
- 运维只能持续手工修配置

## 6. 已完成修复

本轮排障中，已完成以下修复：

### 6.1 修复配置 schema 漂移

已在插件源码中统一配置契约，使 `onboarding` 被显式纳入 plugin manifest schema。

### 6.2 修复运行时路径假设

已将插件运行时的工作目录选择逻辑改为基于以下字段推导：

- `dataDir`
- `clientPath`

从而避免继续依赖旧目录 `/home/openclaw-a2a-mvp`。

### 6.3 修复安装脚本写配置策略

安装脚本已改为：

- 读取 `openclaw.plugin.json` 中的配置字段白名单
- 仅写入 manifest schema 允许的字段
- 避免安装脚本输出超出运行时可接受范围的配置

### 6.4 修复 mvp 安装脚本的持久化缺陷

已修复 `openclaw-a2a-mvp` 安装脚本中“更新名称/策略后未写回配置文件”的问题。

## 7. 整改目标

整改目标不是“修掉一个字段”，而是建立下面四类能力：

- 配置契约一致性
- 安装产物可验证性
- 部署迁移可控性
- 实例运行状态可观测性

只有做到这四点，才能避免未来用户再遇到“几天后自己断掉”的问题。

## 8. 整改方案

### 8.1 第一阶段：立即止血

目标：恢复现有实例，阻止新增实例继续写出非法配置。

措施：

1. 冻结当前会写入非法配置的旧安装脚本版本
2. 批量巡检所有实例的 `~/.openclaw/openclaw.json`
3. 对非法字段进行一次性清理
4. 重启 OpenClaw Gateway 并验证健康状态
5. 对外确认当前推荐安装方式和推荐版本

巡检至少覆盖以下项：

- `openclaw health`
- `http://127.0.0.1:18789/health`
- `plugins.entries.claw-network.config`
- `sidecar_runner` 进程
- backend `/health`
- endpoint 连通性

交付物：

- 一次性实例修复脚本
- 巡检脚本
- 当前推荐安装说明

### 8.2 第二阶段：配置契约收敛

目标：消除 schema 多处定义带来的结构漂移。

措施：

1. 以 `openclaw.plugin.json` 作为唯一配置 schema 来源
2. 运行时代码直接读取 manifest schema，不再手写第二份
3. 安装脚本只能根据 manifest 白名单落盘
4. 所有模板文件也必须与 manifest schema 对齐

要求：

- 禁止在安装脚本中硬编码 schema
- 禁止在运行时代码中维护独立 schema 副本
- 模板文件中的所有字段必须能通过目标 OpenClaw 校验

### 8.3 第三阶段：安装后可启动性校验

目标：把故障拦在发布前，而不是拦在用户重启后。

新增发布前 smoke test：

1. 创建临时 OpenClaw home
2. 使用安装脚本生成配置
3. 校验生成配置是否只包含 schema 允许字段
4. 使用目标 OpenClaw 版本执行 config validate / health
5. 启动临时 Gateway
6. 验证本地健康端点

判定原则：

- 任何一步失败，阻止发布

建议加入 CI 的最小断言：

- 插件可加载
- 安装脚本可运行
- 输出配置合法
- Gateway 可启动

### 8.4 第四阶段：配置迁移机制

目标：允许旧实例平滑升级，不再依赖人工清理字段。

需要引入：

- `configVersion`
- migration runner
- 旧字段清理逻辑
- 版本升级后的自动校验

建议迁移流程：

1. 读取现有配置
2. 判断版本
3. 应用迁移
4. 删除废弃字段
5. 修正路径与默认值
6. 运行配置校验
7. 校验成功后才写回

迁移必须覆盖：

- 旧字段删除
- 字段重命名
- 路径迁移
- endpoint 默认值更新
- onboarding / policy 默认值兼容

### 8.5 第五阶段：状态可观测性建设

目标：将“没反应”拆分为可定位的失败阶段。

建议最少建设以下状态模型：

- `config_invalid`
- `gateway_down`
- `sidecar_down`
- `backend_unreachable`
- `awaiting_connection_approval`
- `awaiting_collaboration_approval`
- `message_delivered`
- `message_timed_out`
- `session_completed`
- `session_failed`

建议建设一条统一自检命令，例如：

```bash
claw-network doctor
```

至少输出：

- OpenClaw 版本
- Gateway 状态
- 插件配置是否合法
- 本地 sidecar 是否运行
- backend 是否可达
- 当前 endpoint 是否可达
- 关键路径是否存在

### 8.6 第六阶段：目录与运行时去硬编码

目标：降低迁移、复制部署、多环境运行时的脆弱性。

要求：

- 运行目录必须从配置推导
- 默认路径只能作为兜底，不可作为真相
- 安装脚本必须根据实际项目根目录生成路径
- 文档中的路径示例必须明确标注“示例路径”

禁止：

- 在运行时代码中硬编码单一项目根目录
- 假设所有部署都位于同一台机器、同一目录结构

## 9. 任务清单

### 9.1 当天必须完成

- 完成所有已知实例的配置修复
- 验证所有关键实例 `openclaw health`
- 验证所有关键实例 Gateway 端口
- 固定推荐安装版本与文档
- 暂停使用旧安装脚本

### 9.2 一周内完成

- 完成安装后 smoke test
- 完成批量巡检脚本
- 完成配置迁移脚本
- 增加一条统一自检命令
- 建立最小 CI 校验链路

### 9.3 一个月内完成

- 引入 `configVersion`
- 完成正式 migration 框架
- 建立任务/协作状态模型
- 完成部署拓扑文档与故障分层文档
- 将运维修复从手工操作转为脚本化

## 10. 验收标准

整改完成后，应满足以下验收条件：

### 10.1 配置层

- 插件 manifest、运行时代码、安装脚本不存在 schema 漂移
- 安装脚本输出配置不会触发 OpenClaw config invalid
- 旧版本配置可被自动迁移到合法状态

### 10.2 运行层

- 新实例安装完成后可直接通过 `openclaw health`
- 已升级实例重启后不会因配置问题掉线
- 工程目录迁移后不会因旧路径残留导致运行失败

### 10.3 运维层

- 可以批量巡检实例健康状态
- 可以快速区分配置问题、gateway 问题、backend 问题、审批问题
- 可以在 5 分钟内定位大多数“龙虾不工作”问题的故障层级

## 11. 推荐工程原则

后续所有 `claw-network` 相关开发，应遵守以下原则：

1. 配置 schema 只能有一个事实来源
2. 任何写配置的代码都必须按 schema 白名单输出
3. 任何发布都必须验证“安装后可启动”
4. 任何字段变更都必须附带 migration 方案
5. 任何路径都应通过配置推导，而不是硬编码
6. 任何跨实例协作都应有明确状态，而不是只表现为“有没有回复”

## 12. 最终结论

本次故障不是 OpenClaw 核心产品本身的不稳定，而是 `claw-network` 插件工程约束不足导致的配置漂移事故。

如果只修单个字段，不补：

- schema 单一来源
- 安装后校验
- migration 机制
- 状态可观测性

则未来仍然可能以不同形式复发。

本次整改的重点不应停留在“修掉 `onboarding` 字段”，而应转向：

“把 `claw-network` 从一个能跑的插件，升级为一个可验证、可迁移、可运维的工程系统。”
